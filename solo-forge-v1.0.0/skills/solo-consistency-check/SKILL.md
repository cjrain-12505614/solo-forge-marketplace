# 정합성 교차검증 (solo-consistency-check)

여러 산출물을 만들다 보면 요구사항에 있는데 구현엔 없거나, 스펙과 코드가 어긋나거나, 테스트가 특정 요구사항을 안 덮는 **틈**이 생긴다. 1인 개발은 이 교차검증을 대신 해줄 사람이 없어서 오히려 자동 검증의 가치가 크다. 이 스킬은 **산출물 간 추적성**을 grep 조인으로 점검한다.

## ⚠️ 읽기 전용 원칙 (load-bearing — 이 스킬의 핵심)

검증은 **수정과 분리**한다. 이 검증을 수행할 때는:
- **Read · Grep · Glob 만 사용**한다. **Edit · Bash(파일 변경)로 산출물을 고치지 않는다.**
- 발견한 불일치는 "보고"할 뿐, 이 스킬이 직접 수정하지 않는다(수정은 사용자 판단 후 별도 작업).
- 진짜 독립성이 필요하면 `solo-agent-dispatch`로 **읽기전용 검증자 서브에이전트**(도구 Read/Grep/Glob, 모델 opus 권장)를 띄워 격리 실행한다 — 검증자가 산출물을 만든 주체와 분리될수록 편향이 준다.

## 추적키 (grep 조인의 축)

산출물마다 항목에 **추적키**를 달아두면 grep으로 조인된다. solo-forge 산출물 기준 권장 키:

| 키 | 의미 | 주로 있는 곳 |
|---|---|---|
| `REQ-NN` | 요구사항 | 기능스펙(`docs/02_기획/`), CLAUDE.md |
| `AC-NN` | 수용 기준(Acceptance Criteria) | 기능스펙, QA 시나리오 |
| `SCR-NN` | 화면 | 화면설계, 기능스펙 |
| `TSK-NN` | 작업 | 계획, 커밋 메시지 |
| `TC-NN` | 테스트 케이스 | QA 시나리오, 테스트 코드 |
| `API-NN` | API 계약 | 개발설계, 구현 |

> 키 체계가 아직 없으면, 먼저 기능스펙·QA에 키를 부여하는 것부터 제안한다(키가 있어야 조인이 된다).

## 실행 순서

1. **검증 대상 수집** — Glob으로 관련 산출물 경로를 모은다(기능스펙·화면설계·QA·구현 코드·의사결정). 대상이 지정되면 그것만.
2. **키 인벤토리** — 각 산출물에서 추적키를 grep으로 추출한다. 키별로 "어느 산출물에 나타나는가"를 표로 만든다.
3. **조인 & 갭 판정** — 키를 축으로 산출물을 조인해 아래를 찾는다:
   - **미구현**: 요구사항(REQ/AC)에 있는데 구현/코드에 대응 키 없음
   - **미검증**: 요구사항/기능에 대응하는 테스트(TC) 없음
   - **고아**: 구현/테스트에 있는데 어느 요구사항에도 안 걸리는 항목
   - **모순**: 같은 키가 산출물 간 다른 값/동작을 서술
4. **리스크 분류** — 발견을 High / Medium / Low로 나눈다.
5. **보고** — RTM(추적 매트릭스: 키 × 산출물) + 갭 목록 + 리스크로 보고. **수정하지 않는다.**

## 출력 형식

```
정합성 교차검증 — {대상}

## 추적 매트릭스 (키 × 산출물)
| 키 | 기능스펙 | 화면 | QA(TC) | 구현 |
|----|:---:|:---:|:---:|:---:|
| REQ-01 | ✓ | ✓ | ✓ | ✓ |
| REQ-02 | ✓ | ✓ | ✗ | ✓ |   ← 미검증

## 갭 (High/Medium/Low)
- [High] REQ-05: 기능스펙엔 있으나 구현·테스트 모두 없음 (미구현)
- [Med] TC-11: 어느 요구사항에도 안 걸림 (고아 테스트)
- [Low] API-03: 스펙 "PATCH" vs 구현 "PUT" (모순)

## 판정: 정합 / 갭 N건 (수정 권고)
```

## 원칙

- **검증자는 산출물을 만들지 않는다.** 발견만 보고하고 수정은 넘긴다.
- 키가 없어 조인이 불가하면, "정합함"이 아니라 "추적 불가 — 키 부여 필요"로 정직하게 보고한다.
- 근거는 grep 히트(파일:줄)다. 인상으로 "일관돼 보인다"고 말하지 않는다.
