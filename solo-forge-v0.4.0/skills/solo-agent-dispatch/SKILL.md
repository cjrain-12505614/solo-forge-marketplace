# 에이전트 배치 스킬

이 스킬은 팀 에이전트들을 효율적으로 배치하여 프로젝트 작업을 병렬 또는 순차로 처리합니다. 각 에이전트는 명확한 목표를 가지고 독립적으로 작동합니다.

## 1. 유연한 팀 구조

### 1.1 고정 멤버

| 코드 | 이름 | 역할 | 설명 |
|------|------|------|------|
| **PL** | 준호 | 프로젝트 리드 | 모든 에이전트의 조율자, 최종 결정권 (Claude) |
| **PM** | 민수님 | 프로젝트 매니저 | 구현 담당자 (사용자), 최종 산출물 소유자 |

### 1.2 에이전트 풀 (프로젝트 규모에 따라 선택)

| 역할 | 담당 업무 | 산출 위치 | 관련 스킬 |
|------|---------|---------|---------|
| **리서처** | 기술/시장 조사, 요구사항 분석, 경쟁사 분석 | `docs/01_참고자료/` | agent-dispatch |
| **기획자** | 요구사항 명세, 사용자 흐름, 기능 설계 | `docs/02_기획/` | agent-dispatch |
| **디자이너** | UI/UX 설계, 와이어프레임, 시각 설계 | `docs/03_설계/UI_와이어프레임/` | agent-dispatch |
| **아키텍트** | 기술 아키텍처, 시스템 설계, 인프라 설계 | `docs/03_설계/시스템아키텍처/` | agent-dispatch |
| **검증자** | 테스트 케이스 작성, QA, 버그 리포팅 | `docs/05_검증/` | agent-dispatch |
| **FE개발자** | 프론트엔드 구현, UI 컴포넌트 | `docs/04_개발/src/frontend/` | agent-dispatch |
| **BE개발자** | 백엔드 구현, API, 데이터 처리 | `docs/04_개발/src/backend/` | agent-dispatch |
| **DevOps** | 배포, 인프라, CI/CD, 모니터링 | `docs/04_개발/` | agent-dispatch |
| **데이터 분석가** | 데이터 모델, 성능 분석, 최적화 | `docs/01_참고자료/` | agent-dispatch |
| **테크라이터** | 기술 문서, API 문서, 사용자 매뉴얼 | `docs/06_산출물/` | agent-dispatch |

### 1.3 팀 규모 기준

**🟢 소규모** (5-15 기능, 1-2개월):
- PL + PM + 2-3 에이전트
- 권장: 리서처, FE/BE개발자 (또는 풀스택 개발자)

**🟡 중규모** (15-50 기능, 3-4개월):
- PL + PM + 4-5 에이전트
- 권장: 리서처, 기획자, 디자이너, FE개발자, BE개발자
- 선택: 아키텍트, 검증자

**🔴 대규모** (50+ 기능, 6개월 이상):
- PL + PM + 5명 이상 에이전트
- 모든 역할 포함: 리서처, 기획자, 디자이너, 아키텍트, FE개발자, BE개발자, DevOps, 검증자, 데이터 분석가, 테크라이터

### 1.4 현재 팀 확인

```
CLAUDE.md - Section 2에서 팀 구성 확인
```

확인 내용:
- PL(준호) ✓
- PM(민수님) ✓
- 투입된 에이전트 이름과 역할 ✓
- 각 에이전트의 전문 분야 ✓

## 2. 배치 규칙 (a10-project 기반)

### 2.1 배치 원칙

**원칙 1: 명확한 목표 하나**
```
❌ "요구사항 분석과 기술 조사를 해줘" → 2개 목표 (분산)
✅ "요구사항 분석을 해줘" → 1개 목표 (명확)
```

**원칙 2: 작업 분할**
```
복잡한 큰 작업을 에이전트 수만큼 분할
예: "전체 API 설계" → 에이전트 A: 사용자 API, 에이전트 B: 데이터 API
```

**원칙 3: 병렬화**
```
독립적인 작업은 동시에 투입
예: 리서치와 기획은 병렬 가능, 개발과 테스트는 순차
```

**원칙 4: 투입 체크리스트**
- [ ] CLAUDE.md Section 2 (팀 구성) 확인
- [ ] 각 에이전트의 전문 분야 확인
- [ ] decision_log.md 최신 항목 추출
- [ ] 산출물 저장 위치 사전 확인
- [ ] "계획 수립이나 확인 요청 없이 바로 실행하세요" 포함

### 2.2 배치 우선순위 결정

1. **현재 Phase 확인**: CLAUDE.md Section 8 (로드맵)
2. **팀 가용성 확인**: 각 에이전트의 현재 상태
3. **의존성 확인**: decision_log.md에서 선행 결정사항
4. **우선순위**: 우선순위 높은 태스크부터 배치

## 8. 에이전트 결과 수집 및 보고

### 8.1 결과 수집

각 에이전트가 산출물 생성 후:

1. **산출물 확인**: 지정된 위치에 저장되었는지 확인
2. **품질 검수**: 성공 기준 충족 확인
3. **메타데이터**: 파일명, 경로, 버전 확인
4. **의존성 확인**: 다음 에이전트가 참고할 부분 체크

### 8.2 PL(준호) 통합

모든 에이전트의 결과를 수집 후:

1. **결과 통합**: 산출물들 간 일관성 확인
2. **의사결정 필요 여부**: 새로운 결정사항 필요하면 decision-tracker 호출
3. **PM 보고 준비**: 세션 종료 보고용 정보 정리

### 8.3 PM 보고 형식

```
PM, 에이전트 배치 결과를 정리했습니다.
```

## 9. 트러블슈팅

### 상황 1: 에이전트가 계획 수립을 요청함

```
문제: 에이전트가 "계획을 먼저 수립해도 될까요?" 라고 요청

해결:
1. 프롬프트를 다시 읽고 "계획 수립이나 확인 요청 없이 바로 실행하세요" 강조
2. 명확한 성공 기준 추가
3. 필요하면 세부 지침 추가
4. 재투입: "다음과 같이 정정하여 바로 진행하세요"
```

### 상황 2: 에이전트의 산출물이 기대 이하

```
문제: 요구사항 명세가 너무 간단하거나 누락이 많음

해결:
1. 문제 부분 구체적으로 지적
2. 세부 지침 추가 (예: "각 기능마다 성공 조건 포함")
3. 참고 자료 제공 (유사 문서)
4. 재투입: "다음 부분을 추가/개선해주세요"
```

### 상황 3: 병렬 투입한 에이전트들의 산출물이 불일치

```
문제: 기획자와 아키텍트의 결과가 상충

해결:
1. 두 에이전트를 불러 협의 유도
2. decision-tracker로 최종 결정
3. 결정 결과를 decision_log.md에 기록
4. 양쪽 에이전트에 반영 요청
```

### 상황 4: 에이전트가 decision_log.md를 무시

```
문제: 이미 결정된 기술을 다시 제안함

해결:
1. 해당 D-번호를 명시적으로 지적
2. "이 결정은 이미 D-XX에서 확정되었습니다" 강조
3. decision_log.md 내용을 프롬프트에 다시 포함
4. 재투입
```

## 10. 에이전트 관리 팁

**팁 1: 한 번에 하나의 명확한 목표**
- "API 설계 및 구현" ❌
- "API 설계" ✅

**팁 2: decision_log.md는 항상 최신**
- 투입 전 2-3번 읽기
- 상충하는 결정이 없는지 확인

**팁 3: 산출물 버전 관리**
- v1, v1.1, v2 형식
- 이전 버전도 보관 (히스토리)

**팁 4: 병렬 투입할 때 협력 포인트 명시**
- "API 계약은 decision_log.md D-XX 참고"
- 두 에이전트가 같은 자료 참고하도록

**팁 5: 에이전트 간 순서 의존성 명확히**
- "기획자 완료 후에 투입하세요" (순차)
- "동시 진행 가능합니다" (병렬)
