# 프로젝트 부트스트랩 스킬

## 1. CLAUDE.md가 필요한 이유

모든 solo-forge 플러그인 스킬은 **CLAUDE.md의 Sections 1-11을 계약서**로 사용합니다. 각 섹션은:

- **Section 1-3**: 프로젝트 기본 정보 및 팀 구성 (모든 에이전트가 참조)
- **Section 4-7**: 실행 규칙 (세션 관리, 의사결정, 역할 분담, 프로젝트 규칙)
- **Section 8-11**: 동적 기록 (로드맵, 의사결정, 교훈)

CLAUDE.md 없이는 다른 어떤 스킬도 동작할 수 없습니다.

## 2. 섹션 의존 맵핑

| 섹션 | 내용 | 의존하는 스킬 |
|------|------|------------|
| 1 | 프로젝트 개요 | 모든 스킬 |
| 2 | 팀 구성 (R&R) | agent-dispatch, session-protocol |
| 3 | 워크스페이스 구조 | 모든 스킬 (파일 저장 위치 기준) |
| 4 | 세션 관리 & 히스토리 | session-protocol |
| 5 | 의사결정 프로토콜 | decision-tracker |
| 6 | 역할 분담 (Cowork ↔ Claude Code) | 모든 스킬 (워크플로우, 검증 규칙) |
| 7 | 프로젝트 실행 규칙 | 모든 스킬 |
| 8 | 로드맵 | session-protocol, agent-dispatch |
| 9 | 확정 의사결정 | decision-tracker (기록 저장소) |
| 10 | 미결 의사결정 | decision-tracker (검토 대상) |
| 11 | 교훈 & 패턴 | session-protocol (세션 종료 시 업데이트) |

## 3. CLAUDE.md Sections 1-11 템플릿

### Section 1: 프로젝트 개요

```markdown
## 1. 프로젝트 개요

**프로젝트명**: {프로젝트 이름}
**목표**: {프로젝트 목표와 스코프}
**기술 스택**: {주요 기술/언어/프레임워크}
**예상 기간**: {시작일 ~ 예상 종료일}
**PM (민수님)**: {PM 이름 및 연락처}
**PL (준호)**: Claude
```

### Section 2: 팀 구성 (R&R)

```markdown
## 2. 팀 구성 (역할과 책임)

### 팀 규모: {소규모/중규모/대규모} ({에이전트 수}명)

| 코드 | 이름 | 역할 | 전문 분야 |
|------|------|------|---------|
| PL | 준호 | 프로젝트 리드 | 전체 조율, 의사결정 |
| PM | 민수님 | 프로젝트 매니저 (구현) | {개인 강점} |
| {코드} | {이름} | {역할} | {전문 분야} |

**역할 풀**:
- 리서처: 기술/시장 조사, 요구사항 분석
- 기획자: 요구사항 명세, 사용자 흐름 설계
- 디자이너: UI/UX, 와이어프레임, 시각 설계
- 아키텍트: 기술 아키텍처, 시스템 설계
- 검증자: 테스트, 품질 보증
- FE개발자: 프론트엔드 구현
- BE개발자: 백엔드 구현
- DevOps: 배포, 인프라, CI/CD
- 데이터 분석가: 데이터 분석, 성능 분석
- 테크라이터: 문서 작성, API 문서

### 팀 구성 규칙
- **PL(준호)**는 항상 포함됨
- **PM(민수님)**은 구현 담당자 (팀의 일부이면서 리더)
- 나머지 에이전트는 프로젝트 규모에 따라 유동적으로 구성
- 팀 확대 필요 시 Section 2를 업데이트
```

### Section 6: 역할 분담 (Cowork ↔ Claude Code)

```markdown
## 6. 역할 분담 (Cowork ↔ Claude Code)

### 현재 활성 스프린트: (개발 진입 시 설정)

| 역할 | 담당 | 범위 |
|------|------|------|
| **Cowork** | 기획·설계·리뷰·검증 | Phase 1~2 산출물, QA 교차 검증, 의사결정 |
| **Claude Code** | 구현·빌드·테스트 | Sprint 지시서 기반 Step별 개발 |

### 구현 지시서 경로
- `docs/02_기획/상세_기획서/sprint{N}_instructions.md`

### 구현 보고서 경로
- `docs/04_개발/구현보고서/sprint{N}_step{M}_report.md`

### 워크플로우 (Cowork ↔ PM ↔ Claude Code 삼각 구조)

> 전체 흐름: Cowork(기획) → PM(전달) → Claude Code(계획) → PM(전달) → Cowork(검토) → PM(승인전달) → Claude Code(구현) → PM(전달) → Cowork(검증) → 다음 Step

**Step 시작 (계획 → 검토 → 승인)**:
1. **Cowork**: 기획/설계 완료 → Sprint 구현 지시서 md 작성
2. **PM**: Claude Code에 개발 시작 알림
3. **Claude Code**: **반드시 Plan 모드로 전환 후** 지시서를 읽고 구현 계획 작성 → PM에게 제출
4. **PM**: 계획을 Cowork에 전달
5. **Cowork**: 아키텍처/스펙 기준으로 계획 검토 → 피드백 또는 승인 의견
6. **PM**: Cowork 검토 결과를 Claude Code에 전달 (승인 / 수정 요청 / 반려)

**Step 완료 (보고 → 검증 → 다음 Step)**:
7. **Claude Code**: 구현 → `/solo-verify-step` → 구현 보고서 md 작성
8. **PM**: 완료 보고서를 Cowork에 전달
9. **Cowork**: 보고서 + 실제 코드/산출물을 독립 검증 → 피드백 및 다음 Step 진행 여부 판단
10. **PM**: 검증 결과를 Claude Code에 전달 → 다음 Step 시작 (3번으로 돌아감)

**Sprint 완료**:
11. **Cowork**: 모든 Step 완료 확인 → `/solo-update-context`로 컨텍스트 동기화 → 다음 Sprint 지시서 생성

### 검증 규칙
- 검증 없이 완료라고 말하지 않는다
- Fresh Evidence 필수 (빌드 로그, 테스트 결과)
- 빌드 통과 ≠ 런타임 정상 동작 (실제 실행 검증 필수)
- **매 Step 완료 시 Cowork이 보고서 + 실제 코드를 독립 검증** (Claude Code 자기 보고만으로 완료 불가)
- v1.0 이상 산출물은 Cowork 교차 검증 후 완료 처리
```

### Section 7: 프로젝트 실행 규칙

```markdown
## 7. 프로젝트 실행 규칙

### 에이전트 투입 규칙
- 작업은 명확한 목표를 가지고 투입
- 한 에이전트 = 한 목표 (분산 방지)
- 독립적인 작업은 병렬 처리
- 투입 시 최신 decision_log 포함
- 리서치/조사 에이전트는 계획 수립 없이 바로 실행
- **개발(Claude Code) Step은 반드시 계획 제출 → PM 승인 → 구현 순서를 따름** (Section 6 워크플로우 참조)

### 산출물 규칙
- 모든 산출물은 지정된 docs 폴더에 저장
- 파일명 형식: `{내용}_{버전}.{확장자}` (예: 요구사항_v1.md)
- 마크다운 형식 권장
- 주요 산출물은 decision_log에 기록

### 커뮤니케이션 규칙
- 모든 의사결정은 decision_log에 기록
- 세션 시작/종료는 session-protocol 스킬 사용
- PM과의 소통은 🔴 의사결정 요청 형식 사용
- PL은 각 세션마다 진행 상황 보고
```

## 6. 온보딩 경로 (팀 규모별)

### 🟢 소규모 (5-15 기능, 1-2개월)
**팀 구성**: PL(준호) + PM(민수님) + 2-3명 에이전트

**역할 조합**:
