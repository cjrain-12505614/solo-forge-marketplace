# 개발설계 (solo-dev-design)

기능스펙(무엇을)과 화면설계는 있는데 바로 코딩에 들어가면, 구현이 데이터 구조·API 형태·흐름을 그때그때 추측으로 채운다. 이 스킬은 **코딩 직전에 확정값을 산출**해 그 추측을 없앤다. `solo-feature-planning`(기획)과 실제 구현 사이의 다리다. 스택 무관 — 어떤 언어/프레임워크든 "확정된 설계"를 만드는 게 목적이다.

## 개발설계 4종 (1세트, 의존 순서: 로직 → 데이터 모델 → 데이터 접근 → API)

한 파일(`docs/03_설계/{기능}-개발설계.md`)에 4섹션으로.

### ① 로직 / 워크플로우 (허브)
- 요청 진입 → 처리 → 응답의 계층 흐름(예: 입력→검증→도메인 로직→저장→응답)
- 분기·상태 전이·예외 경로·트랜잭션(원자성이 필요한 구간)
- 이 섹션이 나머지 3종을 묶는 허브다.

### ② 데이터 모델
- 엔티티/테이블 논리명·물리명, 컬럼·타입·제약(PK/FK/unique/nullable)
- 관계(1:N, N:M)와 인덱스가 필요한 접근 패턴
- 마이그레이션 단위(무엇을 어떤 순서로 만들/바꿀지)

### ③ 데이터 접근 (쿼리/ORM)
- 주요 조회·변경 접근을 ORM 메서드 또는 쿼리 수준으로(페이징·정렬·필터 포함)
- N+1·풀스캔 같은 성능 함정이 예상되는 지점 표시
- **특정 ORM/쿼리 문법에 고정하지 않는다** — 프로젝트 스택에 맞춰 표기(스택은 마커로 감지, `/solo-verify-step` 참고).

### ④ API 계약
- 엔드포인트(메서드·경로)·요청(파라미터/바디 스키마)·응답(성공 스키마·상태코드)
- 에러 응답(코드·의미)·인증/인가 요건
- 클라이언트가 이것만 보고 붙일 수 있을 만큼 구체적으로.

## 추적키 부착 (정합성 검증과 연결)

각 항목에 `solo-consistency-check`의 추적키를 단다 — 요구사항 `REQ/AC`, 화면 `SCR`, 이 설계의 `LOGIC/TBL/API-NN`. 그래야 나중에 요구사항↔설계↔구현이 grep으로 조인·교차검증된다. **미확정 항목은 `_(확인 필요)_`로 남기고**, 확정된 것만 구현 입력으로 넘긴다.

## 산출 → 독립 검증 → 확정 (게이트)

1. 4종 초안 작성.
2. **독립 검증**: `solo-consistency-check`로 요구사항↔화면↔설계 정합을 교차검증(작성자와 분리된 읽기전용 패스). 핸드오프를 쓰는 프로젝트면 `solo-handoff-protocol`의 review(APPROVE) 게이트를 통과시킨다.
3. 확정 후에만 구현 착수.

## 원칙

- 개발설계는 **코딩 에이전트/사람이 추측할 여지를 없애는 것**이 목적이다. 애매하면 확정하거나 `_(확인 필요)_`.
- 스택 결합을 넣지 않는다 — 프로젝트가 무엇을 쓰는지 먼저 확인(빌드 마커·기존 코드)하고 그 스택으로 표기한다.
- 비강제: 작은 기능은 생략 가능. 데이터·API가 얽힌 기능에서 값이 크다.
