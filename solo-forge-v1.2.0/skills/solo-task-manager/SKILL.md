# 작업 관리자 (solo-task-manager)

지금 무엇이 진행 중이고 무엇이 남았는지, 무엇이 막혀 있는지를 **파일로 지속**되게 관리한다. 호스트의 TaskList(이 세션 한정)와 달리 `docs/00_관리/tasks.md`에 남아 세션이 바뀌어도 이어진다. `solo-feature-planning`(무엇을 만들지)·`solo-session-protocol`(세션 연속성)과 함께 쓰면 작업 맥락이 끊기지 않는다.

## 작업 파일

`docs/00_관리/tasks.md` (없으면 생성):
```
# 작업 보드 — {프로젝트}

## 진행 중
- [T-01] 인증 API 구현 · 개발중 · 목표 2026-07-20
  - 블로커: 없음

## 예정
- [T-02] 비밀번호 재설정 · 검토중

## 블로커
- [T-03] 결제 연동 · 외부 키 발급 대기 (의존: T-01)

## 최근 완료 (최근 3개월)
- [T-00] 프로젝트 스캐폴딩 · 완료 2026-07-13
```

## 상태 값

| 상태 | 의미 |
|---|---|
| 검토중 | 무엇을·가능한지 따지는 단계 |
| 진행중 | 구현/작업 중 |
| 검증 | 테스트·리뷰 중 |
| 완료 | 끝 |
| 보류 | 일정 미정·우선순위 하향 |

## 출력 순서 (현황 요약 시 이 순서로)

1. **진행 중** — 지금 작업 중(진행중·검증)
2. **예정** — 다음 할 일(검토중)
3. **블로커** — 진행을 막는 것(의존성·외부 대기)
4. **최근 완료** — 최근 3개월

## 충돌 감지

지금 논의/작업 중인 것과 직접 부딪히는 항목이 있으면 별도로 강조한다.
```
⚠️ 충돌 주의
- 진행 중: [T-01] 사용자 스키마에 필드 추가
- 예정:   [T-05] 사용자 스키마 재설계
→ 같은 대상을 건드림. 순서/우선순위 조정 필요.
```

## 세션 재사용 원칙

같은 주제의 작업이 이미 진행 중이면 새로 만들지 말고 그 항목을 이어간다. 새 작업 추가 전 `tasks.md`에서 유사·중복 항목을 먼저 확인한다.

## 호스트 TaskList와의 관계

- **호스트 TaskList** = 이 세션 안에서의 즉석 실행 추적(세션 종료 시 사라짐).
- **solo-task-manager(`tasks.md`)** = 세션을 넘는 지속 보드.
- 큰 흐름은 `tasks.md`에, 세션 안 잘게 쪼갠 실행 단계는 호스트 TaskList에 두면 둘이 상보한다. 작업 완료 시 `/solo-update-context`로 상태를 반영한다.

## 원칙

- 담당자 필드는 1인 프로젝트에선 생략(팀이면 추가 가능).
- 상태는 실제 근거로 갱신한다(완료는 검증 통과 후).
