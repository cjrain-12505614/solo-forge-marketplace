# 문서 큐레이터 (solo-document-curator)

문서가 쌓인 폴더(다운로드·초안·유입 폴더)를 정리한다. 파일명을 표준화하고, `v1/v2/v3`처럼 섞인 버전을 묶어 최종본을 고르고, 구버전을 삭제후보로 분리한다. **핵심 안전장치: 절대 먼저 지우지 않고, 근거와 함께 후보로만 제안한다.**

## 기본 원칙 (안전 우선)

1. 파일을 **먼저 지우지 않는다** — 삭제는 근거를 붙여 후보로 제시.
2. 같은 문서군을 묶고 최종본 후보를 먼저 고른다.
3. 날짜는 가능하면 **문서 내부 히스토리의 마지막 수정일** 기준(메타데이터 날짜는 참고만).
4. 내부 히스토리 확인이 안 되면 보류로 둔다.
5. 파일명 변경·삭제는 반드시 근거와 함께 제안한다.

## 작업 순서

### 1. 폴더 인벤토리
대상 폴더를 훑어 파악: 파일 수, 확장자별 분포, 버전 표기 유무, 같은 문서군 후보, 최종본/구버전 후보. (`ls -la` + 파일명 패턴 분석; PDF·docx는 내부 수정일 확인 시 텍스트 추출)

### 2. 날짜 기준 확인
- PDF: 내부 히스토리 표의 마지막 수정일 우선(`pdftotext`로 추출, 이미지형이면 Read로 시각 확인).
- 확인 불가 시 파일 메타 날짜는 참고만 남기고 **보류**.

### 3. 표준 파일명
프로젝트에서 한 번 정한 규칙을 폴더 전체에 일관 적용. 권장 기본형:
```
YYMMDD_[프로젝트] 기능또는대상_문서종류_v버전.확장자
예) 260317_[가온] 채번설정_화면설계_v3.0.pdf
```
프로젝트 접두·문서종류 어휘·기능명 표기는 한 번 정하면 통일한다.

### 4. 분류

| 분류 | 조건 |
|---|---|
| 보관본 | 단건 문서 또는 최종 버전 |
| 삭제후보 | 같은 문서군 + 상위 버전 존재 + 구버전 |
| 보류 | 제목은 비슷하나 내용 차이 의심 / 히스토리 미확인 |

### 5. 최종 정리 보고
- 검토한 폴더 경로 / 문서군 요약
- `현재 파일명 → 제안 파일명` 목록
- 삭제후보 목록(**근거 포함**) / 보류 목록 / 추가 확인 필요 문서

## 출력 규칙

- 삭제는 바로 실행하지 않고 후보로 제시한다(사용자 확인 후 삭제).
- 날짜 기준을 못 확인한 문서는 그 사실을 명시한다.
- 정리(파일명 표준화) 후 자산화(문서 내용을 컨텍스트/기획 문서로 전환)로 넘어가는 전처리 스킬이다.
