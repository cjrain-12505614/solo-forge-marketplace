# 히스토리 관리자 (solo-history-manager)

기능이 "지금 어떻게 되어 있나"(코드가 말해준다)가 아니라 **"왜 이렇게 되었나"**를 보존한다. 시간이 지나면 당시의 제약·트레이드오프·요청 배경이 잊혀서, 나중에 "이거 왜 이렇게 짰지?" 하며 같은 실수를 되풀이한다. `solo-decision-tracker`(무엇을 정했나)·`solo-lessons-learned`(무엇을 배웠나)와 함께 "왜"의 층을 담당한다.

## 파일 구조

`docs/00_관리/history/`:
```
history/
├── _timeline.md              # 전체 개발 흐름 (가장 먼저 참고)
├── phase-1.md                # 1차 개발 맥락
├── phase-N.md                # N차 …
└── requests/                 # 이해관계자·이슈별 반영 이력
    └── {날짜}-{이슈}-{제목}.md
```

### _timeline.md
- 차수별 시기·주요 변경·배경 요약
- 주요 아키텍처 결정 이력(상세는 phase-N)

### phase-N.md
- 개발 배경·목표 / 주요 변경 기능
- **아키텍처 결정과 그 이유** / 당시 기술 제약 / 알려진 기술 부채

### requests/{날짜}-{이슈}-{제목}.md
- 요청 배경·원문
- **반영된 것 vs 남은 제약을 분리**(미반영 사유 포함)
- 반영 버전·시점

## 사용 원칙

1. **기능 설명이 아니라 "변경 이유"를 남기고 찾는다.**
   - ✗ "필드가 추가됨" → ○ "X 이슈로 이 값이 필요해서 필드 추가, 그 때문에 API 구조 변경"
2. 특정 이슈/이해관계자 요청이 있으면 `requests/`를 먼저 본다.
3. 같은 설계 결정이 반복 등장하면 `_timeline.md`에 상위 요약이 있는지 확인한다.
4. 현재 코드 맥락과 함께 읽어야 해석이 정확하다 — 히스토리는 "왜", 코드는 "무엇".

## 기록 시점

- 개발 차수 완료 → `phase-N.md`
- 이해관계자/이슈 요청 반영 → `requests/` 파일
- 주요 아키텍처 결정 → `_timeline.md` + `phase-N.md`

`/solo-update-context`로 상태를 반영할 때 히스토리도 함께 갱신하면 맥락이 끊기지 않는다.

## 원칙

- "고객사" 대신 프로젝트의 실제 이해관계자(사용자·이슈·본인)로 프레이밍한다.
- 마크다운 참조는 클릭 가능한 상대 링크로.
