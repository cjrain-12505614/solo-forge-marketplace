# 산출물 관리 스킬

## 개요
프로젝트 산출물의 조직, 명명 규칙, 버전 관리, 품질 보증을 일관되게 관리합니다.

## 파일 명명 규칙

### 기본 형식

```
{타입}_{주제}_{선택_순번}_v{버전}.{확장자}

요소 설명:
- {타입}: feature_spec, wireframe, qa_scenario, db_schema, poc, research, session, etc.
- {주제}: auth, payment, user_management, 등 (영문 스네이크_케이스)
- {선택_순번}: 같은 주제 여러 파일 시 001, 002 등 (선택사항)
- {버전}: 시맨틱 버전 (v0.1, v0.5, v1.0, v1.1, v2.0 등)
- {확장자}: md, sql, json, yaml, etc.
```

### 예시

```
feature_spec_auth_v1.0.md        # 인증 기능 스펙, 완료 버전
qa_scenario_payment_v0.5.md      # 결제 QA 시나리오, 검토 중
db_schema_v1.0.sql               # DB 스키마, 확정 버전
poc_real_time_sync_v0.1.ts       # 실시간 동기화 POC, 초기 버전
research_competitor_analysis.md  # 경쟁사 분석 보고서
session_2026-03-23_1.md          # 2026년 3월 23일 첫 번째 세션
design_system_v2.0.md            # 디자인 시스템, 메이저 업데이트
```

## 버전 관리 규칙

### 시맨틱 버전 정의

| 버전대 | 상태 | 설명 | 사용처 |
|--------|------|------|--------|
| v0.1 ~ v0.9 | 초안 (Draft) | 작업 진행 중, 내부용 | 세션 내 공유 |
| v1.0 | 검증 완료 (Reviewed) | 품질 게이트 통과 | 공식 산출물 |
| v1.1 ~ v1.9 | 마이너 개선 (Minor) | 범위 변경 없는 수정 | 부분 갱신 |
| v2.0 이상 | 메이저 변경 (Major) | 스코프/설계 변경 | 전체 갱신 |

### 버전 변경 가이드

```
v0.1 → v0.2: 오탈자, 포맷 수정 (내용 큰 변화 없음)
v0.5 → v0.9: 초안 단계에서의 주요 내용 추가/수정
v0.9 → v1.0: 검증 완료, 품질 게이트 통과 ★
v1.0 → v1.1: 오탈자, 비고 추가 (스코프 유지)
v1.0 → v2.0: 요구사항/설계 변경으로 인한 전체 재작성
```

## v1.0 승격 프로세스 (품질 게이트)

### 1단계: PL 자체 검증

**산출물 유형별 체크리스트**

#### 📋 기능 스펙 (feature_spec_*.md)
- [ ] 모든 요구사항 ID (REQ-XX) 매핑 완료
- [ ] 사용자 스토리 작성 완료 (형식: "As a ... I want ... so that ...")
- [ ] 수용 기준(AC) 정의 (최소 3개 이상)
- [ ] 화면 목록 확인 (04_화면설계 문서 참조)
- [ ] 데이터 모델 정의 (엔티티, 필드, 제약사항)
- [ ] API 엔드포인트 정의 (필요 시)
- [ ] 예외 케이스 작성 (validation 오류, 타임아웃, 네트워크 오류 등)
- [ ] 의사결정 로그 참조 (D-XX 형식)
- [ ] 용어 일관성 확인 (팀 사전 확인)
- [ ] TODO, TBD, FIXME 없음 ✅

#### 📐 설계 문서 (system_architecture_v*.md, db_schema_v*.sql, api_design_v*.md)
- [ ] 개요 및 목표 명확함
- [ ] 다이어그램/시각화 포함
- [ ] 대안 검토 기록 (decision_log.md 참조)
- [ ] 제약사항 명시 (성능, 보안, 확장성)
- [ ] 테스트 전략 포함
- [ ] 모호한 표현 없음
- [ ] 코드 스니펫/예제 최신화
- [ ] 용어 일관성 확인

#### 🖼️ 화면설계 (wireframe_v*.md, screen_flow_v*.md)
- [ ] 모든 기능 화면 포함
- [ ] 화면 흐름도 완성
- [ ] 공통 컴포넌트 정의 (헤더, 푸터, 네비게이션)
- [ ] 반응형/다중 해상도 고려 (필요 시)
- [ ] 접근성 기준(A11y) 검토 (필요 시)
- [ ] 상호작용 명시 (클릭, 드래그, 스크롤 등)
- [ ] 오류/로딩 상태 정의

#### 🧪 QA 시나리오 (qa_scenario_v*.md)
- [ ] 모든 기능 테스트 케이스 포함
- [ ] TC ID 형식 일관성 (TC-{기능}-{번호})
- [ ] 정상 경로 + 예외 경로 커버
- [ ] 우선순위 정의 (P1, P2, P3)
- [ ] 사전 조건 명확함
- [ ] 기대 결과 구체적
- [ ] 재현 가능성 확보

#### 📄 공통 사항 (모든 산출물)
- [ ] 제목, 버전, 작성일, 작성자 포함
- [ ] 목차 또는 섹션 구조 명확
- [ ] 링크/참조 모두 유효
- [ ] 이전 버전 대비 변경사항 기록
- [ ] 관련 산출물 크로스 참조

### 2단계: 검증자 크로스 검증 (선택사항, PM 판단)

**검증자**: PL 또는 별도 검토자
**체크항목**:
- 스펙이 요구사항을 충분히 만족하는가?
- 기술적 실현 가능성이 있는가?
- 화면설계와 스펙이 일치하는가?
- QA 시나리오가 스펙을 충분히 커버하는가?

**결과**:
- ✅ 통과
- 🔄 조건부 통과 (경미한 수정 후 v1.0 승격)
- ❌ 재작성 (v0.x 반복)

---

### 3단계: PM 최종 검수

**검수자**: PM
**체크항목**:
- 비즈니스 요구사항 충족도
- 사용자 경험 적정성
- 일정/범위 적합성
- 위험요소 확인

**결과**:
- ✅ 승격 승인 → v1.0 태그
- 🔄 조건부 → 지정 항목 수정 후 재검수
- ❌ 반려 → v0.x에서 계속

---

### 4단계: 문서 갱신

v1.0 승격이 결정되면:

```markdown
1. 파일명 변경: v0.x → v1.0
2. 문서 헤더 갱신:
   - 버전: v1.0
   - 상태: Approved
   - 검수자: {PM명}
   - 검수일: YYYY-MM-DD
3. decision_log.md에 기록:
   - D-XX | {산출물명} v1.0 승격 | Approved | PM | YYYY-MM-DD
4. session 로그에 기록:
   - ✅ {산출물명} v1.0 승격 (검증 완료)
```

## 산출물 관리 워크플로우

### 생성
```
1. team_plan.md에서 필요한 산출물 확인
2. 적절한 폴더 선택 (00_관리 ~ 06_산출물)
3. 명명 규칙에 따라 파일 생성 (v0.1)
4. 기본 템플릿 적용 (제목, 버전, 작성자, 날짜)
5. 내용 작성 시작
```

### 버전업
```
v0.x → v0.y: 세션 내 수정
  └─ PM 승인 필요 없음

v0.x → v1.0: 정식 산출물 승격
  └─ 위 "v1.0 승격 프로세스" 실행 필수

v1.0 → v1.1: 마이너 개선
  └─ 체크리스트 재실행 권장 (오탈자만 제외)

v1.0 → v2.0: 메이저 재작성
  └─ 전체 검증 재실행
```

### 보관/아카이빙
```
- 최신 버전만 docs/{folder}에 유지
- 이전 버전은 별도 보관소 (docs/archive 또는 git history)
- 최신 버전 링크 명시
```
