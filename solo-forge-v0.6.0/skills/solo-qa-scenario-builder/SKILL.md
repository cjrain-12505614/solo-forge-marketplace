# QA 시나리오 생성 스킬

## 개요
기능 스펙, 화면설계서, 데이터 모델을 기반으로 체계적인 QA 테스트 시나리오를 생성합니다. 정상 경로부터 예외 케이스까지 포괄적으로 설계합니다.

## 출력 위치

QA 시나리오 문서는 **docs/05_검증/** 폴더에 저장됩니다.

```
docs/05_검증/
├── qa_scenario_auth_v1.0.md
├── qa_scenario_payment_v0.5.md
├── test_result_2026-03-23.md
└── issue_log_v1.0.md
```

## 테스트 케이스 ID 형식

```
TC-{기능코드}-{순번}

예:
- TC-AUTH-001 : 가입 - 정상 입력
- TC-AUTH-002 : 가입 - 이메일 중복
- TC-AUTH-003 : 이메일 인증 - 정상
- TC-PAYMENT-001 : 결제 - 정상 결제
- TC-PAYMENT-002 : 결제 - 결제 실패
```

### 기능코드 매핑 예시

| 기능 | 코드 |
|------|------|
| 인증 (Authentication) | AUTH |
| 결제 (Payment) | PAYMENT |
| 사용자 관리 (User Management) | USER |
| 상품 관리 (Product) | PRODUCT |
| 주문 (Order) | ORDER |
| 알림 (Notification) | NOTIF |

## 2. 테스트 케이스 (정상 경로)

### TC-AUTH-001: 가입 - 정상 입력

| 항목 | 내용 |
|------|------|
| **TC ID** | TC-AUTH-001 |
| **기능 그룹** | 인증 (Authentication) |
| **시나리오명** | 새로운 사용자 가입 성공 |
| **우선순위** | P1 (Critical) |
| **관련 화면** | SCR-AUTH-001 (가입 화면) |
| **관련 AC** | AC 1.1 (정상 가입) |

**사전 조건**:
- 서비스 접속 가능
- 미가입 사용자
- 네트워크 정상

**테스트 단계**:

| 단계 | 작업 | 기대 결과 |
|------|------|---------|
| 1 | "가입" 버튼 클릭 | 가입 페이지(SCR-AUTH-001) 로드 |
| 2 | 이메일 입력: user@example.com | 필드에 이메일 입력됨 |
| 3 | 비밀번호 입력: SecurePass123! | 필드에 마스킹된 비밀번호 표시 |
| 4 | 비밀번호 확인 입력: SecurePass123! | 필드에 마스킹된 비밀번호 표시 |
| 5 | "가입" 버튼 클릭 | 로딩 인디케이터 표시 (1초) |
| 6 | (API 응답 대기) | "가입 완료! 이메일 확인 링크를 발송했습니다" 메시지 |
| 7 | 페이지 이동 | 이메일 확인 페이지(SCR-AUTH-002)로 자동 이동 |

**실제 결과**:
- [ ] PASS
- [ ] FAIL (오류 내용 기록)
- [ ] SKIP (건너뛴 이유 기록)

**비고**:
- 계정 생성 확인: DB에서 user@example.com 검색
- 이메일 발송 확인: 이메일 수신함에서 확인 링크 확인

## 3. 테스트 케이스 (예외 케이스)

### TC-AUTH-003: 가입 - 비밀번호 미입력

| 항목 | 내용 |
|------|------|
| **TC ID** | TC-AUTH-003 |
| **기능 그룹** | 인증 (Authentication) |
| **시나리오명** | 필수 필드 미입력 - 비밀번호 |
| **우선순위** | P2 (High) |
| **관련 화면** | SCR-AUTH-001 (가입 화면) |
| **관련 AC** | AC 1.4 (필수 필드 검증) |

**사전 조건**:
- 가입 페이지 접속
- 이메일은 입력, 비밀번호는 공백

**테스트 단계**:

| 단계 | 작업 | 기대 결과 |
|------|------|---------|
| 1 | 이메일 입력: user@example.com | 필드에 입력됨 |
| 2 | 비밀번호 필드는 공백으로 둠 | 필드가 비어있음 |
| 3 | "가입" 버튼 클릭 | 버튼이 비활성화 상태이거나, 클릭 후 오류 메시지 |
| 4 | (오류 메시지 확인) | "필수 항목입니다" 경고 메시지 표시 |
| 5 | 비밀번호 필드 하이라이트 | 빨간 테두리 또는 배경색 강조 |

**실제 결과**:
- [ ] PASS
- [ ] FAIL (오류 내용 기록)
- [ ] SKIP (건너뛴 이유 기록)

**비고**:
- 클라이언트 단에서 검증이 이루어지는지 확인
- API 요청이 발송되지 않는지 확인

- TC-AUTH-003: 필수 필드 미입력

## 기능: 이메일 인증 (Email Verification)
- TC-AUTH-006: 정상 인증
- TC-AUTH-007: 토큰 만료

```

## 8. 우선순위 정의

### P1 (Critical)
- 비즈니스 핵심 기능
- 사용자 데이터 손실 위험
- 보안 결함
- 예: 가입, 로그인, 결제

### P2 (High)
- 중요한 기능
- 사용성 영향
- 예: 비밀번호 검증, 에러 처리

### P3 (Medium)
- 일반적인 기능
- UI 개선사항
- 예: 로딩 인디케이터, 안내 메시지

### P4 (Low)
- 부가 기능
- 사소한 UI 결함
- 예: 색상 미세 조정, 폰트 크기

## 사용 예시

### 예시 1: 기본 요청
**PM**: "가입 기능 테스트 시나리오 만들어줘"
→ 기능 스펙(feature_spec_auth_v1.0.md) 참조
→ qa_scenario_auth_v0.1.md 생성

### 예시 2: AC 기반 생성
**PM**: "이 AC들을 커버하는 테스트 케이스 뽑아줘"
→ AC 목록 분석
→ 각 AC별 TC 최소 1개씩 생성

### 예시 3: 화면 기반 요청
**PM**: "이 화면(SCR-AUTH-001)의 모든 상호작용을 테스트하는 시나리오 만들어줘"
→ 화면설계서 분석
→ 모든 사용자 흐름 커버하는 TC 생성
